# CS73 — Prod-Deploy Cold DB Handling

Closes step toward [CS73](../blob/main/project/clickstops/active/active_cs73_prod-deploy-cold-db-handling.md). Eliminates the cold-pause failure mode from `prod-deploy.yml` so a routine prod deploy after long idle succeeds on the **first** attempt without operator intervention.

## What changed

- **`scripts/wake-db.js`** (new). Standalone Azure SQL "wake" utility used by `prod-deploy.yml` immediately before `Run DB migrations`. Talks to `mssql` directly — does **NOT** import `server/db/mssql-adapter.js` (the runtime adapter's `MssqlAdapter.CONNECT_TIMEOUT_MS = 5000` constant is deliberate per CS53-3 / CS53-6 and must stay 5s for the user-request warmup-retry path).
  - DI shape mirrors `scripts/migrate.js`: `wakeDb({ sql, connectionString, perAttemptTimeoutMs, totalBudgetMs, sleep, log })` + `main()` CLI wrapper.
  - Defaults: `perAttemptTimeoutMs=30_000`, `totalBudgetMs=150_000` (~5 attempts). Both timeouts applied to `connectionTimeout` and `requestTimeout` (top-level + tedious `options.*`) so `SELECT 1` can't fail on a default-15s request timeout while we are still inside a per-attempt budget.
  - Defaults use `??` (not `||`) and validate positive numbers — passing `0` surfaces as a clear validation error rather than being silently replaced.
  - Retries on Azure SQL transient codes `40613, 40197, 40501, 49918, 49919, 49920` and `ConnectionError` / `RequestError`. Linear backoff 5s / 10s / 15s capped.
  - Pool always closed via `finally{}` — and **before** the backoff sleep — so a failed connection is not held open during the retry window (especially important for 49918/19/20 resource-pressure cases).
- **`.github/workflows/prod-deploy.yml`** — two new steps inserted immediately before "Run DB migrations":
  1. **Report DB status before wake (CS73)** — uses the pinned `azure/cli` action (consistent with the rest of the workflow) to log `Initial DB status` via `az sql db show`.
  2. **Wake Azure SQL gwn-production (CS73)** — runs `node scripts/wake-db.js` and logs the elapsed wake time so the deploy log shows whether the run paid the cold-pause cost or was already warm.
- **`tests/wake-db-script.test.js`** (new). 13 cases covering: first-attempt success, retry-then-success on transient `40613`, total-budget exhaustion → exit 1, default + override `connectionTimeout` / `requestTimeout` carried into config, validation rejecting `0`-valued timeout/budget, pool closed on failure path, **close-before-sleep ordering**, non-retryable errors surfaced immediately, `DATABASE_URL` env fallback, and CLI exit codes.
- **`LEARNINGS.md`** — cold-pause section updated with "Resolved by CS73 (PR #330)" note; operator workaround block removed.
- **CS73 plan task table** — CS73-1 through CS73-4 marked ✅ Done.

Validation of the runtime cold-pause behavior against a real paused Azure SQL DB happens **post-merge** on the next prod deploy that organically follows ≥60min idle (Azure SQL `GP_S_Gen5` serverless does not support manual pause). See § Validation strategy in the CS73 plan file.
